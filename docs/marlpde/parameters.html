149def jacobian_sparsity():
150    '''
151    It turns out the the Jacobian sparsity matrix, as provided by
152    previous versions of this function was too strict. The diagonals
153    had a width of 1, this implicitly assumes no dependencies of the
154    Jacobian elements on fields at adjacent depths. A run with such
155    a Jacobian sparsity matrix took a lot of compute power and time and
156    ultimately halted without covering a single time step! A debugging 
157    session with breakpoints inside solve_ivp showed that the Jacobian 
158    matrix computed by `solve_ivp` (when neither the Jacobian nor the 
159    Jacobian sparsity matrix was provided) has non-zero elements not 
160    only on the diagonals but also on adjacent positions along the 
161    diagonals. This was confirmed by ChatGPT:
162    "It is indeed common for the Jacobian matrix of coupled partial 
163    differential equations (PDEs), such as advection-diffusion equations, 
164    to have non-zero elements beyond the main diagonal. This phenomenon, 
165    where adjacent elements in the Jacobian are non-zero, is often due 
166    to the spatial discretization of the differential equations."
167    Likewise computing a functional Jacobian is only feasible for the
168    diagonals, for positions near the diagonals it is very hard.
169
170    Here we will choose diagonals of width 3.
171    '''
172
173    no_depths = Map_Scenario().N
174    # Number of fields = 5, i.e. calcite, aragonite, concentrations of
175    # calcium and carbonate ions and the porosity.
176    no_fields = 5
177    n = no_fields * no_depths
178    
179    diagonals = np.ones((3 * (2 * no_fields - 1), n))
180
181    # Construct the offsets.
182    offsets = ()
183    for offset in range(-n + no_depths, n - no_depths + 1, no_depths): 
184        offsets = offsets + ((offset -1, offset, offset + 1),)
185    # Turn offsets into a single tuple instead of a tuple of tuples.
186    offsets = sum(offsets, ())
187    # Check that we have as many offsets as diagonals:
188    try:
189        assert len(offsets) == diagonals.shape[0]
190    except AssertionError as e:
191        print('Setup of diagonals incorrect.')
192    # Construct the sparse matrix. 
193    raw_matrix = lil_matrix(dia_matrix((diagonals, offsets), shape=(n, n))) 
194    # Set the Jacobian elements to zero that correspond with the derivatives
195    # of the rhs of equations 40 and 41 wrt phi.
196    raw_matrix[:2 * no_depths, 4 * no_depths:] = 0
197
198    return csr_matrix(raw_matrix)

190        assert len(offsets) == diagonals.shape[0]
191    except AssertionError as e:
192        print('Setup of diagonals incorrect.')
193    # Construct the sparse matrix. 
194    raw_matrix = lil_matrix(dia_matrix((diagonals, offsets), shape=(n, n))) 
195    # Set the Jacobian elements to zero that correspond with the derivatives
196    # of the rhs of equations 40 and 41 wrt phi.
197    raw_matrix[:2 * no_depths, 4 * no_depths:] = 0
198
199    return csr_matrix(raw_matrix)

It turns out the the Jacobian sparsity matrix, as provided by previous versions of this function was too strict. The diagonals had a width of 1, this implicitly assumes no dependencies of the Jacobian elements on fields at adjacent depths. A run with such a Jacobian sparsity matrix took a lot of compute power and time and ultimately halted without covering a single time step! A debugging session with breakpoints inside solve_ivp showed that the Jacobian matrix computed by solve_ivp (when neither the Jacobian nor the Jacobian sparsity matrix was provided) has non-zero elements not only on the diagonals but also on adjacent positions along the diagonals. This was confirmed by ChatGPT: "It is indeed common for the Jacobian matrix of coupled partial differential equations (PDEs), such as advection-diffusion equations, to have non-zero elements beyond the main diagonal. This phenomenon, where adjacent elements in the Jacobian are non-zero, is often due to the spatial discretization of the differential equations." Likewise computing a functional Jacobian is only feasible for the diagonals, for positions near the diagonals it is very hard.

Here we will choose diagonals of width 3.
